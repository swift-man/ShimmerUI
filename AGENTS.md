# AGENTS.md

## 목적
이 문서는 ShimmerUI 저장소에서 에이전트가 코드를 수정할 때 준수해야 할 **SOLID 기반 설계 원칙**과 작업 규칙을 정의합니다.

## 전제
- 대상: Swift 기반 UI/컴포넌트 및 데모 코드
- 목표: 읽기 쉬운 구조, 변경에 강한 구조, 테스트 가능한 구조

## S.O.L.I.D 설계 원칙

### 1) SRP - Single Responsibility Principle
- 한 파일(또는 타입)은 하나의 변경 이유만 가져야 합니다.
- Swift 타입(클래스/구조체/열거형/프로토콜)은 UI 렌더링, 상태 관리, 비즈니스 규칙, 저장소 접근 같은 역할을 섞지 않습니다.
- View는 렌더링, ViewModel은 상태/로직, Repository는 데이터 접근으로 분리합니다.

### 2) OCP - Open/Closed Principle
- 기존 동작을 깨뜨리지 않고 새 기능을 추가할 수 있어야 합니다.
- 조건 분기(`if/else`)가 커지는 영역은 프로토콜 기반 전략/컴포저블 구조로 분리합니다.
- `enum`이 과도하게 성장하면 프로토콜 추상화와 디스패치로 확장 경로를 확보합니다.

### 3) LSP - Liskov Substitution Principle
- 파생 타입은 부모 타입의 계약을 깨지 않아야 합니다.
- 상속보다 프로토콜 채택을 우선 사용하고, 구현 계약은 최소한의 불변 조건으로 둡니다.
- 서브타입 교체 시 예측되지 않은 크래시/동작 불일치가 생기지 않도록 사전 조건을 추가 강화하지 않습니다.

### 4) ISP - Interface Segregation Principle
- 클라이언트가 사용하지 않는 API에 의존하지 않도록, 프로토콜은 최소 단위로 분리합니다.
- 거대한 "god protocol" 대신 역할별 프로토콜(`Drawable`, `Animatable`, `Loadable`)을 분해합니다.
- 테스트/플랫폼별 가짜 구현은 필요한 인터페이스만 채택합니다.

### 5) DIP - Dependency Inversion Principle
- 고수준 모듈이 저수준 모듈에 직접 의존하지 않고 추상층(프로토콜)에 의존합니다.
- 객체 생성은 외부 팩토리/DI 구성을 통해 주입합니다.
- `import Foundation` 수준의 공통 유틸만 직접 의존하고, 구체 기능은 인터페이스로 감쌉니다.

## 적용 규칙
1. 모듈 분리
   - `ShimmerUI`는 컴포넌트, 애니메이션 엔진, 렌더링 유틸, 테스트 더블로 구분합니다.
2. 이름 규칙
   - 타입 이름은 역할을 명시적으로 드러내야 합니다. (예: `ShimmerAnimator`, `ShimmerConfigProviding`)
3. 의존 방향
   - 상위 레이어 `App`/`Feature` → `Interface` → `Implementation` 순으로 단방향 의존을 유지합니다.
4. 변경 관리
   - 변경 제안 시 해당 책임을 수정할 때 다른 원칙 위반이 없는지 함께 점검합니다.
5. 테스트 가능성
   - 상태 계산 로직, 애니메이션 스케줄러, 데이터 바인딩 코드는 가능한 순수 타입/함수로 분리해 유닛 테스트 대상화합니다.

## 예시 구조 (권장)
- `Sources/ShimmerUI/Core/` : 공통 기본 유틸
- `Sources/ShimmerUI/Animation/` : 애니메이션 스케줄러 및 시간 추상화
- `Sources/ShimmerUI/Render/` : 렌더링 파이프라인
- `Sources/ShimmerUI/Feature/*` : 화면별 기능 모듈
- `Tests/ShimmerUITests/` : 동작/계약 기반 테스트

## 예외
- 성능 최적화가 즉각적으로 필요한 구간은 우선순위상 가장 단순한 방식으로 시작하되, 원칙 위반이 장기적 유지보수 비용을 키우지 않는지 최소 주기적으로 점검합니다.
